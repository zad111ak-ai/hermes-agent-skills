# Universal Adaptive Loops — Архитектура самообучающихся циклов для Hermes

## Концепция

Лупы (loops) — это механизмы, которые позволяют агенту:
- **Адаптироваться** к изменениям (сайты меняют структуру, провайдеры ломаются)
- **Учиться** на ошибках (запоминать что сработало/нет)
- **Эволюционировать** (повышать качество при тех же мощностях)

Ключевое требование Dima: **лупы должны быть universal** — работать в реальном времени, не через cron, встроенными в рабочий процесс агента.

## Трёхслойная архитектура

### ⚡ Слой 1: Fast Loop (Reactive) — мгновенная реакция

Срабатывает в рамках одного запроса/сессии.

**Примеры:**
- **Adaptive Scraping:** селектор не сработал → пробуем альтернативный → кешируем новую схему
- **Model Routing:** модель вернула ошибку → пробуем fallback → запоминаем что работает
- **Proxy Rotation:** прокси упал → переключаемся → кешируем статус

**Триггеры:** Ошибка, таймаут, невалидный ответ, 429/403
**Хранилище:** Кеш в памяти сессии + `memory()` для долгого хранения
**Токен-цена:** Минимальная (только логика, без LLM)

### 🔄 Слой 2: Medium Loop (Tactical) — тактическая оптимизация

Срабатывает между сессиями или при накоплении N ошибок.

**Примеры:**
- **Schema Evolution:** после N неудач → переобучить селекторы для домена
- **Prompt Tuning:** проанализировать N неудачных промптов → улучшить шаблон
- **Quality Gate:** после генерации → LLM-as-a-Judge → если < 7/10 → регенерировать

**Триггеры:** N ошибок за период, окончание сессии, запуск похожей задачи
**Хранилище:** `learning.py` + `outcomes.jsonl` + `patterns.json`
**Токен-цена:** Умеренная (1-3 LLM вызова на анализ)

### 🧠 Слой 3: Slow Loop (Strategic) — стратегическая эволюция

Срабатывает раз в N часов/дней. Анализирует паттерны, перестраивает стратегию.

**Примеры:**
- **Provider Analysis:** какие провайдеры стабильны? Какие падают? Стоит ли менять приоритеты?
- **Task Pattern Mining:** какие задачи решаются хорошо? Где слабые места?
- **Cost Optimization:** какие модели дешёвые/качественные? Где переплачиваем?

**Триггеры:** Таймер (cron) или накопление N фактов в `memory`
**Хранилище:** `quality_history.db` + `patterns.json`
**Токен-цена:** Высокая (полный анализ N записей), но редкая

## Интеграция с инструментами Hermes

| Инструмент | Роль в циклах |
|-----------|---------------|
| `learning.py` | Fast + Medium — запись исходов, анализ паттернов |
| `quality_loop.py` | Medium — LLM-as-a-Judge оценка качества |
| `provider_loop.py` | Medium/Slow — мониторинг провайдеров, роутинг |
| `memory()` | Все слои — персистентное хранение фактов |
| `skill_manage()` | Slow — запись процедурных знаний |
| `consolidate_memory.py` | Slow — ночная консолидация фактов |

## Пример: эволюция селекторов + запись фактов

```
1. Запрос: harvest scrape https://example.com/products
2. Ошибка: SelectorError: .price не найден
3. Fast Loop: Adaptive Scraping пробует .product-price → успех.
   → Новая схема кешируется в ~/.harvest/schemas/example.com.json
   → Факт в memory: "Adaptive schema updated for example.com: {'price': '.product-price'}"
4. Medium Loop (через 100 ошибок): Self-Learning переобучает схему на всех страницах сайта
   → memory: "Retrained 1 schemas due to 100 errors on example.com"
5. Slow Loop (ежедневно): Анализ — какие сайты чаще ломаются? Стоит ли сменить подход?
```

## Токен-экономия в циклах

1. **Кеширование:** схемы, прокси, промпты → не пересчитывать каждый раз
2. **Дельта-обработка:** только изменения, не полные данные
3. **Квантование:** GGUF для локальных задач вместо API

## Адаптивные задержки

При падении: увеличивать задержку между ретраями.
```
0 провалов → 0s
1 провал  → 30s
2 провала → 120s
3+ провала → 300s + увеличивать на 50% каждый раз
При успехе: сбросить задержку к базовой
```
