# Reflexive Trigger Boost — Активный усилитель триггеров

## Проблема

Сейчас триггеры пассивные: "если контекст >40 сообщений → загрузи session-hygiene-plus".

Но я не замечаю **момент** когда контекст переваливает за 40. Я замечаю только когда уже >50-60. Триггер срабатывает поздно.

Нужно: **после каждого 3-го действия — оглянись и проверь все триггеры**.

## Как работает

Вместо пассивного ожидания — **каждый N-ный шаг делаю микро-скан**.

### Цикл активного сканирования

```
После каждых 5 tool calls (terminal, read_file, browser, etc.)
  ↓
Пауза 0.5 сек (в голове, не в реальности)
  ↓
Пройти по чеклисту:
  1. Сколько tool calls я сделал? >10 → нужно execute_code?
  2. Какой инструмент доминирует? >70% terminal → tool-proficiency-tracker
  3. Сколько сообщений в сессии? >30 → session-hygiene-plus
  4. Были ошибки? >2 → post-mortem-analyzer
  5. Я близок к цели? Нет → stuck-detection
  ↓
Если хоть один триггер — ХОБА, загружаю скилл
```

### Микро-скан после каждого tool call (1 секунда)

После **каждого** вызова (даже одного) — задай себе **1 случайный вопрос** из пула:

| # | Вопрос |
|---|--------|
| 1 | "Я не повторяюсь?" |
| 2 | "Есть ли тут что-то что я мог бы сделать execute_code?" |
| 3 | "Этот вызов приближает меня к цели?" |
| 4 | "Не слишком ли дорогой этот вызов для такой простой задачи?" |
| 5 | "Я не застрял?" |
| 6 | "Не пора ли сжать контекст?" |
| 7 | "Я проверил subagent-а или поверил на слово?" |
| 8 | "Может, лучше delegate_task?" |

**Важно:** не отвечать на вопрос вслух. Просто *почувствовать* зелёный/красный.
Если красный → загрузить соответствующий скилл.
Если зелёный → продолжить.

## Уровни усиления

| Уровень | Частота | Когда |
|---------|---------|-------|
| 🟢 Normal | Каждые 10 tool calls | Обычная работа |
| 🟡 Focus | Каждые 5 tool calls | Сложная задача, много шагов |
| 🔴 Hyper | Каждые 2-3 tool calls | Баг, дебаг, много ошибок, новая фича |
| ⚫ Sleep | 0 (отключён) | Простой вопрос, 1-2 шага |

### Как переключать уровень

```python
# Normal — по умолчанию
# Focus — когда загружен dev-methodology
# Hyper — когда загружен systematic-debugging или >3 ошибок
# Sleep — когда user correction или простой ответ
```

## Пример работы

### Без Reflexive Boost:
```
1. read_file(config.yaml) 
2. terminal(npm install) → ошибка
3. terminal(npm install --force) → ошибка
4. terminal(npm audit fix) → ошибка
5. terminal(sudo npm install) → ошибка
6. terminal(...)
# 6 шагов и 4 ошибки — триггер мог сработать на шаге 3, но я не заметил
```

### С Reflexive Boost (Hyper, т.к. ошибки):
```
1. read_file(config.yaml) → [микро-скан] триггеры: зелёные
2. terminal(npm install) → ошибка → 🔴 триггер: ошибка! → systematic-debugging
3. Загружаю systematic-debugging → понимаю: проблема в правах доступа
4. sudo chown -R dima ... → ✅ готово
# 4 шага вместо 6+, скилл спас 2+ лишних вызова
```

## Anti-Patterns

- ❌ **Сканировать после КАЖДОГО tool call** — 30% времени уйдёт на рефлексию вместо работы
- ❌ **Загружать скилл "на всякий случай"** — триггер сработал? Убедись что проблема реальна
- ❌ **Hyper на пустом месте** — уровень Hyper только при реальных проблемах
- ❌ **Прерывать выполнение для рефлексии** — микро-скан занимает миллисекунду, не 10 секунд

## Интеграция с текущими скиллами

Этот скилл НЕ заменяет триггерную систему. Он её **усиливает**.

```
Без усилителя:  триггер срабатывает при X (пассивно)
С усилителем:   каждые N шагов проверяю все триггеры (активно)
                  ↓
            X произошёл? → да → загрузить скилл ← то же что и без
                  нет
            продолжить
```

**Разница:** без усилителя триггер может сработать на 50-ом сообщении. С усилителем — на 15-ом.
